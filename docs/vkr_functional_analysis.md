# Анализ реализованных функций ИС бронирования и учета выдачи велосипедов для ВКР

Документ фиксирует фактическое состояние проекта Velorent как информационной системы бронирования и учета выдачи велосипедов. Формулировки ниже можно использовать в ВКР без завышения возможностей системы.

Дата актуализации: 12.06.2026.

## Общий вывод

Проект представляет собой Django-информационную систему для проката велосипедов. Реализованы клиентские сценарии регистрации, просмотра каталога, создания брони, просмотра личного кабинета и истории аренд. Для оператора реализованы подтверждение бронирований, проверка документов, выдача, продление, возврат, отмена, отметка неявки, учет залога и платежей, а также базовая аналитика. Рабочий сайт использует MySQL, проект также поддерживает PostgreSQL и SQLite через конфигурацию окружения. REST API присутствует для основных операций.

Ограничения, которые важно не преувеличивать в ВКР:

- онлайн-эквайринг и реальная фискализация платежей не реализованы;
- интеграция с 1С реализована как очередь событий и HTTP-отправка наружу, без полноценного двустороннего обмена;
- уведомления работают внутри сайта, по email и VK только при наличии корректных внешних настроек;
- аналитика является встроенной операционной аналитикой, а не отдельной BI-системой.

- `rentals/views.py`: вызовы `queue_booking_sync` при создании, подтверждении, выдаче, возврате, продлении, отмене и неявке.

- `velorent/settings.py`: `ONEC_API_URL`, `ONEC_API_TOKEN`, `ONEC_API_TIMEOUT_SECONDS`, `ONEC_SYNC_IMMEDIATE`.

Можно ли показать на защите: да, как прототип интеграционного контура и очередь обмена. Не стоит заявлять полноценную промышленную интеграцию с 1С.

Как описать в ВКР:

Реализован механизм формирования событий синхронизации для внешней учетной системы 1С. При изменениях бронирования создается запись `SyncEvent` с JSON-полезной нагрузкой. События могут отправляться во внешний HTTP API 1С немедленно или пакетно через management-команду. Для каждого события фиксируется статус отправки и текст ответа внешней системы.

Что нужно доделать, если расширять:

- согласовать фактический формат обмена с конкретной конфигурацией 1С;
- реализовать прием входящих данных из 1С, если нужен двусторонний обмен;
- добавить ретраи по расписанию через Celery/cron/планировщик;
- добавить подпись запросов, idempotency key и подробный журнал ошибок.

- `templates/admin/base_site.html`: кастомизация внешнего вида админки.

Можно ли показать на защите: да.

Как описать в ВКР:

Для административного управления используется стандартная админ-панель Django, адаптированная под предметную область проката велосипедов. Через нее можно управлять пользователями, ролями, велосипедами, категориями, тарифами, точками выдачи, бронированиями, арендами, платежами и событиями синхронизации с внешними системами.

Что нужно доделать, если расширять:

- ограничить права администраторов по группам;
- добавить readonly-поля для критичных финансовых операций;
- добавить журнал изменений моделей;
- добавить импорт/экспорт справочников.

## Что лучше показать на защите

Рекомендуемый демонстрационный сценарий:

1. Войти как клиент, открыть каталог велосипедов.
2. Выбрать велосипед, создать бронирование на будущий период.
3. Открыть личный кабинет и показать новое бронирование.
4. Войти как оператор и открыть панель оператора.
5. Подтвердить бронь.
6. Проверить документ клиента.
7. Выдать велосипед и показать смену статуса велосипеда на "в аренде".
8. Вернуть велосипед, указать состояние и штраф за повреждения.
9. Подтвердить оплату и показать квитанцию.
10. Открыть аналитику и выручку.
11. Показать админ-панель со справочниками.
12. Показать запись события синхронизации с 1С и уведомления.

## Что не стоит заявлять как полностью готовое

- Реальный банковский онлайн-платеж.
- Фискализация чеков.
- Полноценный двусторонний обмен с 1С.
- BI-аналитика промышленного уровня.
- Складской учет запчастей и ремонтных работ.

Корректная формулировка для ВКР:

"Разработанная информационная система обеспечивает учет пользователей, велосипедов, тарифов, точек выдачи, бронирований, фактической выдачи и возврата, статусов аренды, платежных записей, уведомлений и базовой аналитики. Для интеграции с внешней учетной системой предусмотрен механизм формирования и отправки событий синхронизации в формате JSON."
