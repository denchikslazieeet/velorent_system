# Системное проектирование VeloRent

## 1. Назначение системы
Система предназначена для автоматизации бизнес-процессов велопроката:
- ведение каталога велосипедов;
- онлайн-бронирование;
- выдача и возврат;
- учет залогов, платежей и штрафов;
- контроль технического состояния;
- синхронизация с 1С.

## 2. Архитектурный подход
Выбран стиль **modular monolith**:
- одна кодовая база на Django;
- отдельные прикладные модули по доменам;
- REST API для внешних клиентов и интеграций;
- PostgreSQL как основная БД;
- отдельный интеграционный слой для 1С.

Такой подход лучше всего подходит для небольшого и среднего бизнеса: проще внедрение, меньше стоимость поддержки, легче масштабировать команду и код.

## 3. Доменная модель
### Справочники
- PickupLocation — точки выдачи;
- BikeCategory — категории велосипедов;
- Tariff — правила расчета аренды.

### Операционные сущности
- Bike — единица велопарка;
- Booking — предварительное бронирование;
- Rental — фактическая аренда;
- Payment — денежные операции;
- SyncEvent — журнал обмена с 1С.

## 4. Роли пользователей
### Клиент
- просматривает каталог;
- бронирует велосипед;
- видит свои аренды и историю.

### Оператор
- подтверждает бронь;
- выдает велосипед;
- принимает возврат;
- фиксирует состояние и платежи.

### Администратор
- управляет справочниками;
- контролирует операторов;
- проверяет интеграцию и отчеты.

## 5. Жизненный цикл аренды
1. Клиент выбирает велосипед.
2. Создает бронь на период.
3. Система проверяет пересечения и рассчитывает стоимость.
4. Оператор подтверждает бронь.
5. При выдаче создается активная аренда и принимается залог.
6. При возврате рассчитываются итоговая стоимость, просрочка и штрафы.
7. Возврат залога фиксируется отдельной платежной операцией.
8. Данные отправляются в 1С.

## 6. Почему не микросервисы
Для курсовой и реального малого бизнеса микросервисная архитектура избыточна:
- сложнее отладка и деплой;
- дороже поддержка;
- слабее учебная и практическая целостность.

Поэтому модульный монолит здесь — зрелое и оправданное решение.

## 7. Предлагаемое развитие
- вынести интеграцию с 1С в очередь сообщений;
- подключить Celery для фоновых задач;
- добавить объект «Пункт возврата» отдельно от точки выдачи;
- реализовать фотофиксацию состояния велосипеда;
- внедрить push/SMS/Telegram-уведомления.
